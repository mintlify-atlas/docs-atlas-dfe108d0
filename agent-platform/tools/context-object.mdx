---
title: "Context object"
description: "The central runtime state container for workflow execution—capturing inputs, node states, outputs, and metadata."
---

The Context Object is the central runtime state container for every workflow execution. It updates dynamically as the workflow progresses and serves as the shared memory layer that nodes read from and write to.

## What it tracks

| Category | Key fields | Description |
|---|---|---|
| **Run metadata** | `startDate`, `status`, `flowType`, `appId`, `versionId`, `cfProcessId` | Identifiers and timestamps for the workflow instance |
| **Execution state** | `steps`, `branch`, `activeSteps`, `status` | Each node's progress and outcome |
| **Environment and config** | `env`, `agentTimeout`, `asyncExecDetails` | Runtime and environment parameters |

## The `steps` object

`steps` is the most critical part of the Context Object—a map indexed by node name or ID that holds the state of each node in the workflow.

Each node entry contains:

| Field | Description |
|---|---|
| `input` | Data received by the node at execution time |
| `output` | Result produced by the node |
| `statusCode` / `isSuccessful` | Whether execution succeeded, failed, or was skipped |
| `logs` | Log text or debug messages generated by the node |
| `executionTime`, `totalTime` | Node-level timing data |
| `reqObj` / `resObj` | For model-invoking nodes: raw request and response envelopes |
| `metadata` | Node-specific information such as schema references or task IDs |

### Example node structure

```json
"steps": {
  "ExampleNode": {
    "input": {
      "parameters": { "param1": "value1" }
    },
    "output": {
      "result": "some value",
      "summary": "optional explanation"
    },
    "statusCode": 200,
    "isSuccessful": true,
    "logs": "Node executed successfully",
    "executionTime": "1250ms"
  }
}
```

## Using the context object

### Cross-node referencing

Nodes reference each other's data through context interpolation using the `{{ }}` syntax. Values are resolved at runtime.

```json
"steps": {
  "SchemaRetriever": {
    "output": { "schema": [ ... ] }
  },
  "QueryGenerator": {
    "input": {
      "schema": "{{steps.SchemaRetriever.output.schema}}"
    }
  }
}
```

### Reference patterns

```
{{context.appId}}
{{context.steps.QueryGenerator.output.sql_query}}
{{context.schema[0].column_name}}
```

### Evaluation behavior

- **Lazy resolution** — Values are resolved immediately before each node executes.
- **Missing keys** — Resolve to `null` (or an empty string in string contexts).
- **Circular references** — Automatically detected and blocked.

<Note>Typing `{{` in any field that supports context variables displays a dropdown of all available variables, grouped by node—including environment variables defined at the workflow level.</Note>

## Loop node context

When a loop node executes, it creates its own context object tracking the loop's progress.

```json
{
  "loopId": "string",
  "startTime": "timestamp",
  "iterationCount": "number",
  "inputs": ["array of input values"],
  "outputArray": ["array of iteration results"],
  "totalCount": "number",
  "childNodesExecuted": "number"
}
```

### Accessing loop data

**Inside the loop:**

| Variable | Description |
|---|---|
| `{{context.currentItem}}` | The item currently being processed |
| `{{context.currentIndex}}` | Zero-based index of the current item |
| `{{context.currentOutput[x]}}` | Output from a previous iteration |

**Outside the loop:**

| Variable | Description |
|---|---|
| `{{context.steps.Loop0001.output}}` | Complete array of all iteration results |
| `{{context.steps.Loop0001.output[x]}}` | A specific iteration's result |

## Extending with script nodes

Script nodes let you extend the Context Object programmatically during execution.

<CodeGroup>
```javascript JavaScript
// Add a simple key-value pair
context.newKey = "Hello World";

// Store structured data
context.customData = {
  runId: context.cfProcessId,
  stage: "validation",
  timestamp: new Date().toISOString()
};
```

```python Python
# Add or modify keys in the context dictionary
context['newKey'] = "Hello World"

# Store complex objects
context['customData'] = {
    "runId": context.get('cfProcessId'),
    "stage": "validation",
    "timestamp": datetime.now().isoformat()
}
```
</CodeGroup>

### Guidelines

- Use camelCase naming (`customerInfo`, `retryCounter`, `tempResults`)
- Do not override reserved keys: `steps`, `branch`, `status`, `dbquery`
- Keep data lightweight—avoid storing large arrays or raw API responses
- All keys added to the context are accessible downstream via `{{context.<key>}}`

## Best practices

1. Use consistent node naming for traceability (for example, `QueryGenerator`, `DataCleaner`).
2. Mask sensitive data before writing it to the context.
3. Prefer declarative references (`{{ ... }}`) for readability whenever possible.
4. Use script nodes only when dynamic or computed context manipulation is required.
