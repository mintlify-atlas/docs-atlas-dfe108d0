title: "Workflow nodes"
description: "Reference for all node types available in AI for Process workflows, including AI, agent, API, logic, control, human, utility, and scanner nodes."
---

Nodes are the building blocks of a workflow. Each node performs a specific operation — running an AI model, calling an API, routing execution, or handling human tasks — and passes its output to the next node through the workflow context.

## How nodes perform operations

When a workflow runs, each node receives input from the workflow context, processes it, and writes its output back. Downstream nodes reference that output using the `{{context.steps.NodeName.output}}` syntax.

| Field | Description |
|---|---|
| `input` | Data received by the node at execution time. |
| `output` | Result produced by the node. |
| `statusCode` / `isSuccessful` | Whether execution succeeded, failed, or was skipped. |
| `logs` | Debug messages generated during execution. |
| `executionTime` | Time taken to complete the node. |

## Node types at a glance

| Category | Nodes | Purpose |
|---|---|---|
| **Control** | Start, End | Define entry and exit points for the workflow. |
| **AI** | Text-to-Text, Text-to-Image, Image-to-Text, Audio-to-Text, Doc Intelligence, Agentic App | Run AI model tasks across text, images, audio, and documents. |
| **Integration** | API, Function, DocSearch | Connect to external systems and run custom code. |
| **Logic** | Condition, Loop, Split, Merge | Control branching, iteration, and parallel execution. |
| **Human** | Human | Route tasks to people for review, approval, or input. |
| **Utility** | Delay, Variable, Log | Pause execution, transform variables, and add logging. |
| **Scanner** | Input Scanner, Output Scanner | Validate prompts and responses using guardrail policies. |

## Control nodes

### Start node

Every workflow begins with a Start Node. It defines how the workflow is triggered and validates the input schema. Configure triggers — API call, schedule, or event — directly in this node.

### End node

The End Node terminates the workflow and returns the final output. Map outputs from any upstream node to the End Node's output schema.

## AI nodes

AI nodes are multimodal components powered by LLMs and AI models. Each node processes inputs and generates outputs that downstream nodes can reference.

### Text-to-Text

Generates, transforms, or analyzes text using an LLM.

Use cases: summarization, content generation, translation, classification, entity extraction.

### Text-to-Image

Generates images from text prompts.

Use cases: marketing visuals, product mockups, AI-driven design workflows.

### Image-to-Text

Extracts text or descriptions from images using vision models.

Use cases: OCR, image captioning, visual QA, content moderation.

### Audio-to-Text

Converts spoken audio to written text using OpenAI Whisper-1. Supports transcription in multiple languages and translation to English.

- **Supported formats:** M4a, MP3, Webm, MP4, Mpga, Wav, Mpeg.
- **File size limit:** 25 MB maximum. Split larger files at logical points.
- **Translation:** Transcribes and translates non-English audio to English. Reverse translation is not currently supported.
- **Token limit:** Whisper processes up to 224 tokens in the input prompt. Input exceeding this limit is ignored.

Use cases: meeting transcription, customer support automation, subtitle generation, voice command processing.

### Doc Intelligence

Extracts structured data from documents using OCR and AI models.

Use cases: invoice processing, form extraction, contract analysis, receipt processing.

### Agentic App node

Integrates a deployed Agentic App into the workflow. When the workflow reaches this node, input is passed to the Agentic App, which interprets the task, performs the required processing, and returns output for downstream nodes.

**Key behaviors:**
- Performs one turn of communication per execution. For multi-step interactions, add multiple Agentic App Nodes in sequence.
- The Agentic App must be deployed and must belong to the same workspace as the workflow.
- Output is accessible at `{{context.steps.NodeName.output}}`.

Use cases: incident triage, invoice review and routing, complex decisions on unstructured data.

## Integration nodes

### API node

Calls external REST or SOAP APIs. Supports synchronous and asynchronous execution, flexible authentication, and custom headers and payloads.

| Type | Behavior | Timeout range |
|---|---|---|
| **Synchronous** | Waits for response before proceeding. | 5–180 seconds (default: 60s). |
| **Asynchronous** | Continues processing without waiting. | 30–300 seconds (default: 60s). Use **No timeout** for long-running processes. |

**Body formats:** `application/json`, `application/xml`, `application/x-www-form-urlencoded`, Custom.

**Authorization options:**
- **Pre-authorize**: Use a system-level token already authorized in advance.
- **Allow users to authorize**: Each user provides their own credentials at runtime.

Access API node output using: `{{context.steps.Start.APINodeName}}`.

Use cases: data enrichment, document retrieval, webhook triggers, external notifications.

### Function node

Runs custom code within the workflow.

Supported runtimes: Python 3.9+, Node.js 18+.

For available Python libraries, see [Supported libraries](/ai-for-process/supported-libraries).

### DocSearch node

Searches a document repository and returns relevant results based on a query.

## Logic nodes

### Condition node

Routes workflow execution based on whether defined conditions are met. Supports `IF`, `ELSE IF`, and `ELSE` paths with `AND`/`OR` logic.

Conditions support context variables (`{{context.variable}}`), previous node outputs, and static values.

<Note>A Condition Node can be called a maximum of 10 times in a single workflow run. Exceeding this limit results in an error.</Note>

### Loop node

Iterates over a collection, executing a set of nodes for each item. The loop creates its own scoped context object tracking iteration progress and results.

### Split node

Executes multiple branches in parallel. Supports waiting for all branches to complete before proceeding.

### Merge node

Combines results from parallel branches into a single output for downstream processing.

## Human node

Routes a task to a human for action. The workflow pauses at this node until the assigned person completes the task, then resumes.

**Task types:**

| Type | Description |
|---|---|
| **Review** | Review and edit AI-generated output. |
| **Approval** | Approve or reject a workflow decision. |
| **Input** | Provide missing information required to continue the workflow. |
| **Classification** | Categorize or label data for downstream processing. |

Configure assignees by role or specific user, set a timeout and escalation action, and define fields the human can view or edit.

## Utility nodes

### Delay node

Pauses workflow execution for a fixed duration or until a scheduled time.

### Variable node

Sets or transforms variables in the workflow context.

### Log node

Logs messages and variable values for debugging at info, debug, warn, or error levels.

## Scanner nodes

Input and output scanners validate prompts and responses using guardrail policies configured in your workspace. Input scanners evaluate what is sent to an LLM node; output scanners evaluate what the LLM returns.

**Prerequisite:** Scanners must be deployed before you can add them to a workflow.

**To add a scanner:**

<Steps>
  <Step title="Open Guardrails">
    Open your workflow and click **Guardrails** in the left navigation.
  </Step>
  <Step title="Add input scanners">
    In the **Input Scanners** section, click **Add Scanner**, select scanners from the list, and click **Done**.
  </Step>
  <Step title="Configure">
    Click a scanner to configure its settings.
  </Step>
  <Step title="Add output scanners">
    Repeat for output scanners as needed.
  </Step>
</Steps>

**Common scanner settings:**

| Scanner | Settings |
|---|---|
| **Toxicity** | Risk score threshold; option to end flow if threshold is exceeded. |
| **Regex** | Patterns to ban, match type, end flow if threshold is exceeded. |

## Error handling

### Try-catch pattern

Use try-catch blocks to handle node failures and route execution based on the error type. Configure catch conditions such as timeout, validation error, or a default failure action.

### Retry configuration

Configure retry behavior including maximum attempts, delay strategy (fixed or exponential), initial delay, maximum delay, and which error types are retryable (timeout, rate limit, server error).

**Timeout precedence:** Workflow timeout > Node timeout > Model timeout.
